## 비관적 / 낙관적 / 분산 락 비교 및 선택 근거

열무마켓의 동시성 제어 대상은 크게 두 가지이다.

- 첫째, 같은 상품에 여러 사용자가 동시에 주문을 생성하는 흐름이다.
- 둘째, 이미 생성된 주문에 대해 결제, 취소, 배송, 환불, 리뷰 생성처럼 여러 도메인이 주문 상태를 변경하는 흐름이다.

두 문제는 보호해야 하는 자원과 충돌 방식이 다르므로 하나의 락으로 통일하지 않고 역할을 분리했다.

| 구분 | 관리 주체 | 보호 범위 | 성능 특성 | 적합한 시나리오 | 프로젝트 적용 |
| --- | --- | --- | --- | --- | --- |
| DB 비관적 락 | DB 트랜잭션 / JPA `PESSIMISTIC_WRITE` | 특정 DB row | 트랜잭션 동안 row와 DB 커넥션을 점유한다. 강한 직렬화가 쉽지만 락 대기와 데드락 부담이 있다. | 짧은 트랜잭션 안에서 단일 row를 반드시 선점해야 하는 경우 | 주문 상태 변경 축에서는 제거했다. 단, 채팅방 생성처럼 범위가 좁은 상품 조회에는 제한적으로 남겨 두었다. |
| DB 낙관적 락 | DB version column / JPA `@Version` | version이 있는 단일 엔티티 | 선점 락을 잡지 않아 평상시 성능 부담이 낮다. 충돌은 flush/commit 시점에 감지하므로 충돌 후 예외 처리 또는 재시도가 필요하다. | 충돌 빈도는 낮지만 동일 자원의 중복 변경을 막아야 하는 경우 | `Product.@Version`으로 주문 생성 시 동일 상품 중복 예약을 감지한다. 충돌 시 `ORDER_ALREADY_EXISTS`로 변환한다. |
| Redis 분산 락 | Redis / Redisson / 애플리케이션 락 키 | Redis key로 정의한 논리 자원 | DB row 락 대기 중 커넥션을 점유하지 않고, wait time과 lease time을 명시적으로 제어할 수 있다. 대신 Redis 장애와 락 lease, 트랜잭션 경계를 애플리케이션이 관리해야 한다. | 여러 도메인 또는 여러 인스턴스가 같은 논리 자원을 동일 기준으로 직렬화해야 하는 경우 | 기존 주문의 상태 변경은 `lock:order:{orderId}` 공통 키로 직렬화한다. |

### 락 최종 선택

주문 생성은 `Product.@Version` 기반 낙관적 락을 유지하고 이미 존재하는 주문의 상태 변경은 Redis 분산 락으로 통일했다. 즉, 낙관적 락과 분산 락은 대체 관계가 아니라 역할이 다르다.

- 주문 생성: 같은 상품을 여러 구매자가 동시에 주문할 수 있으므로 `Product`의 version 충돌로 중복 예약을 막는다.
- 주문 상태 변경: 결제, 주문 취소, 배송 등록, 구매 확정, 환불, 리뷰 생성은 모두 같은 주문을 기준으로 직렬화해야 하므로 `lock:order:{orderId}` 하나의 공통 키를 사용한다.

초기에는 주문, 결제, 환불 처리에서 DB 비관적 락을 사용했지만, `Order`는 여러 도메인이 공유하는 공통 자원이다. 주문 서비스는 주문 row를 잠그고,
결제 서비스도 주문 row를 잠그고, 환불은 환불 row와 주문 row를 함께 잠그는 방식이면 "같은 주문을 같은 기준으로 직렬화한다"는 정책이 코드에서 잘 드러나지 않는다.

Redis 분산 락으로 전환하면서 이 기준을 `LockKeys.order(orderId)` 한 곳으로 고정했다.
주문, 결제, 환불, 리뷰 생성 흐름은 모두 `DistributedLockExecutor`를 통해 같은 키를 획득한 뒤 상태 변경 트랜잭션을 실행한다.
이로써 도메인별 구현은 달라도 동시성 제어 기준은 주문 단위로 일관된다.

분산 락 도입 후에는 Redisson의 고정 lease 문제도 함께 보완했다.
`tryLock(waitTime, leaseTime)` 방식은 watchdog 자동 연장이 없기 때문에, 락 안의 트랜잭션이 lease보다 오래 걸리면 커밋 전에 락이 만료될 수 있다.
이를 막기 위해 락 내부 상태 변경 메서드에는 `@LockBoundedTransactional`을 적용하고, `LockProperties`가 애플리케이션 기동 시 `tx-timeout-seconds < lease-time` 불변식을 검증한다.
운영 설정은 트랜잭션 timeout 8초, lease 10초로 유지해 락 만료 전에 트랜잭션이 커밋되거나 롤백되도록 강제한다.
