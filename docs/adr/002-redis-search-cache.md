# ADR-002: Redis 기반 검색 캐시와 인기 검색어 집계

- 상태: 채택됨
- 날짜: 2026-06-22
- 관련: `docs/PRD.md` (3장 성공 기준 "검색 성능", 7.3 상품 검색), `docs/API.md` (검색 API), `docs/ERD.md` (Redis 집계 제외)

## 맥락

PRD의 P0 성공 기준에는 같은 조건의 상품 검색에서 반복 조회 성능 개선을 확인하고, 인기 검색어 상위 목록을 제공하는 요구가 있다.
상품의 원본 데이터는 MySQL이 정본이지만, 반복 검색 결과와 인기 검색어 집계는 관계형 테이블만으로 처리하면 성능 개선 여부를 분리해 검증하기 어렵다.

## 검토한 대안

- **MySQL만 사용** — 인프라가 단순하다. 다만 반복 검색 성능 개선과 인기 검색어 집계가 일반 조회 부하와 섞인다.
- **애플리케이션 로컬 캐시** — 도입이 쉽다. 다만 프로세스 재시작 시 집계가 사라지고 서버가 늘어나면 캐시와 인기 검색어가 인스턴스마다 달라진다.
- **Redis** — 별도 인프라가 필요하다. 대신 TTL 기반 검색 캐시와 sorted set 기반 인기 검색어 집계를 분리해서 관리할 수 있고 Spring Data Redis로 다룰 수 있다.

## 결정

검색 결과 캐시와 인기 검색어 집계는 Redis를 사용한다.
애플리케이션 의존성은 `spring-boot-starter-data-redis`를 사용한다.

상품 검색은 정규화한 검색 조건을 캐시 키로 사용하고, 캐시 미스 시 MySQL에서 조회한다.
인기 검색어는 상품 검색 요청의 키워드를 Redis에 집계한 파생 데이터로 관리한다.
MySQL은 상품, 주문, 채팅 등 영속 데이터의 정본으로 유지하고 Redis 데이터는 관계형 ERD에 포함하지 않는다.

## 결과

- 로컬 실행 시 MySQL datasource와 함께 Redis 설정이 필요하다.
- 상품 등록, 수정, 삭제, 숨김처럼 검색 결과에 영향을 주는 변경은 관련 검색 캐시를 무효화해야 한다.
- Redis 장애 또는 캐시 미스 시 상품 검색은 MySQL 조회로 동작해야 한다.
- 인기 검색어는 파생 데이터이므로 영속 데이터 정합성 판단 기준으로 사용하지 않는다.
