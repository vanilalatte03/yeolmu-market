# ADR-001: 동시 주문 제어 방식

- 상태: 채택됨
- 날짜: 2026-06-22
- 관련: `docs/PRD.md` (3장 성공 기준 "동시 주문", 7.4 주문), `docs/API.md` (주문 생성)

## 맥락

PRD 핵심 성공 기준: 하나의 판매 중(`ON_SALE`) 상품에 동시에 여러 주문이 들어와도 성공 주문은 1건만 생겨야 한다.
주문이 성공하면 상품은 `RESERVED`가 되고, 이후 주문은 `PRODUCT_NOT_ON_SALE` 또는 `ORDER_ALREADY_EXISTS`로 거절된다.

문제는 경쟁 상황이다. 두 요청이 동시에 `ON_SALE`을 읽으면 둘 다 검증을 통과해 주문 2건이 생길 수 있다.
이를 막을 방법을 정해야 한다.

## 검토한 대안

- **낙관적 락 (`@Version`)** — 상품 row에 버전 컬럼. 충돌 시 두 번째 요청은 재시도 없이 실패 처리. 추가 인프라 불필요. 충돌이 드문 중고거래 특성에 적합.
- **비관적 락 (`SELECT ... FOR UPDATE`)** — 주문 시 상품 row를 잠금. 확실하지만 동시 접근이 몰리면 대기 발생.
- **DB UNIQUE 제약** — 활성 주문(상품당 1건)에 유니크 제약을 걸어 중복 자체를 차단. 락 없이 보장하나 "활성 주문" 상태 모델링이 필요.

## 결정

상품 row에 JPA 낙관적 락(`@Version`)을 적용한다.
주문 생성 시 주문 row 생성과 상품 상태 변경(`ON_SALE` -> `RESERVED`)을 하나의 트랜잭션에서 처리하고, 같은 상품을 대상으로 한 동시 주문 충돌은 재시도하지 않고 실패 처리한다.

중고거래 상품은 단건 거래이고 같은 상품에 주문이 동시에 몰리는 상황은 드물다. 충돌이 발생해도 둘 중 하나만 성공해야 하므로 서버가 자동 재시도해 얻는 이득이 작고, 재시도 정책을 추가하면 트랜잭션 흐름과 장애 처리가 불필요하게 복잡해진다.

비관적 락은 충돌이 드문 P0 주문 생성 흐름에 비해 대기와 잠금 비용이 크므로 선택하지 않는다.
DB UNIQUE 제약은 보조 방어막으로는 유효하지만, 활성 주문 상태 모델링과 취소/재주문 규칙까지 함께 설계해야 하므로 P0 기본 결정에서는 제외한다.

## 결과

- `Product` 엔티티에 버전 컬럼을 둔다.
- 주문 생성 서비스는 상품을 조회한 뒤 반드시 상품 상태를 `RESERVED`로 변경한다. 단순히 상품을 조회하고 주문만 저장하면 상품 row 버전이 증가하지 않아 낙관적 락이 동작하지 않는다.
- 주문 생성과 상품 상태 변경은 같은 트랜잭션에서 처리한다. 낙관적 락 충돌로 트랜잭션이 실패하면 주문 row도 함께 롤백되어야 한다.
- 낙관적 락 충돌은 `409 Conflict`로 응답한다. 현재 API 에러 모델에서는 `PRODUCT_NOT_ON_SALE` 또는 `ORDER_ALREADY_EXISTS` 중 주문 생성 실패 사유에 맞는 코드로 변환한다.
- 동시 주문 테스트는 같은 상품에 여러 주문 요청이 동시에 들어와도 성공 주문은 1건이고, 실패 요청은 재시도 없이 409로 끝나며, 최종 상품 상태가 `RESERVED`임을 검증한다.
