# ADR-004: Redis 기반 JWT 로그아웃 블랙리스트

- 상태: 대체됨 (ADR-007)
- 날짜: 2026-06-23
- 관련: `docs/PRD.md` (7.1 회원), `docs/API.md` (인증 API), `docs/ERD.md` (Redis 데이터 제외)

## 맥락

현재 로그아웃은 클라이언트가 보관 중인 access token을 삭제하는 방식으로 설명되어 있다.
하지만 JWT access token은 stateless 구조라 서버가 별도 폐기 상태를 관리하지 않으면, 로그아웃 후에도 만료 전 토큰이 다시 사용될 수 있다.

팀은 로그아웃 이후 토큰을 즉시 무효화하는 동작을 API 계약에 포함하기로 결정했다.
이미 프로젝트는 검색 캐시와 인기 검색어 집계를 위해 Redis를 사용하므로, 만료 시간이 있는 토큰 폐기 상태도 Redis TTL과 잘 맞는다.

## 검토한 대안

- **클라이언트 토큰 삭제만 사용** — 구현이 가장 단순하다. 다만 탈취되었거나 다른 저장소에 남은 access token은 만료 전까지 계속 유효하다.
- **MySQL 폐기 토큰 테이블** — 서버가 폐기 상태를 영속적으로 확인할 수 있다. 다만 모든 인증 요청에서 관계형 DB 조회 또는 별도 캐시 동기화가 필요하고, 만료 토큰 정리 작업이 추가된다.
- **Redis 블랙리스트** — 서버 인스턴스 간 폐기 상태를 공유할 수 있고 TTL로 JWT 만료 시점에 맞춰 자동 제거할 수 있다. 대신 인증 경로에 Redis 조회가 추가된다.

## 결정

로그아웃 요청에 사용된 JWT access token은 Redis 블랙리스트에 등록한다.
블랙리스트 항목의 TTL은 해당 JWT의 만료까지 남은 시간으로 설정한다.
TTL이 끝나면 Redis가 블랙리스트 항목을 자동 제거한다.

인증 필터는 JWT 서명과 만료 여부를 검증한 뒤 Redis 블랙리스트 등록 여부를 확인한다.
블랙리스트에 등록된 토큰은 만료 전이라도 인증 실패로 처리한다.

Redis 키에는 원문 access token을 저장하지 않는다.
구현 시 원문 토큰의 해시 또는 `jti` 같은 토큰 식별자를 키로 사용한다.
MySQL은 회원과 거래 데이터의 정본으로 유지하고, 로그아웃 블랙리스트는 관계형 ERD에 포함하지 않는다.

## 결과

- 로그아웃은 클라이언트 로컬 삭제만으로 완료되지 않고 서버 블랙리스트 등록까지 포함한다.
- 같은 access token으로 로그아웃을 반복 호출하면 이미 폐기된 토큰으로 보아 인증 실패가 발생할 수 있다.
- 인증이 필요한 모든 API는 `REVOKED_TOKEN` 상태를 공통 401 에러로 반환할 수 있다.
- 검색 캐시와 달리 로그아웃 블랙리스트는 인증 보안 경로에 있으므로 Redis 조회 실패를 단순 캐시 미스로 취급하지 않는다.
