# ADR-010: 레코드 상태 변경 동시성 제어를 Redis 분산 락으로 통일

- 상태: 대체됨 (ADR-013)
- 날짜: 2026-06-29
- 관련: ADR-001(주문 생성 동시성), `docs/CONVENTIONS.md`("과한 설계 제한"), ADR-013(본 ADR을 대체)

## 맥락

주문 *생성*의 동시성은 ADR-001에 따라 `Product.@Version` 낙관적 락으로 처리한다. 이 결정은 유지한다.

그 외 "이미 존재하는 레코드의 상태를 동시에 바꾸는" 흐름은 현재 DB 비관적 락(`@Lock(PESSIMISTIC_WRITE)` + `findById...ForUpdate`)에 의존한다. 여기서 핵심은 **`Order` row 락이 한 도메인에 속하지 않고 여러 도메인이 공유하는 공통 직렬화 자원**이라는 점이다.

| 잠그는 대상 | 잠그는 코드(호출부) | 직렬화하려는 것 |
| --- | --- | --- |
| `Order` row | `OrderService`(취소·확정), `PaymentService`, `RefundService`, `ReviewService` | 한 주문에 대한 동시 작업(결제·취소·확정·환불·리뷰)을 **도메인 횡단**으로 직렬화 |
| `Payment` row | `PaymentService` | 결제 상태 전이 직렬화 (중복 결제 자체는 멱등키 `findByIdempotencyKey`가 별도 차단) |
| `RefundRequest` row | `RefundService` | 환불 요청 상태 전이 직렬화 |

문제는 셋이다.

1. 한 흐름 안에 생성(낙관)·상태변경(비관) 락이 섞여 있어, 어느 락이 무엇을 막는지 파악하기 어렵다(리뷰에서 "낙관·비관이 복잡하게 섞여 있다"고 지적됨).
2. 비관 락은 트랜잭션이 끝날 때까지 DB row와 커넥션을 점유한다. 트랜잭션이 길어지면 락 보유 시간과 커넥션 점유가 함께 길어진다.
3. 특히 `Order` 락은 네 개 도메인이 같은 row를 잠그는 공통 자원인데, 도메인마다 비관 락 메서드가 흩어져 있어 "같은 주문을 같은 기준으로 직렬화한다"는 보장이 코드에 드러나지 않는다.

## 검토한 대안

- **현행 유지(DB 비관 락)** — 추가 인프라가 없고, 트랜잭션이 락 생명주기를 원자적으로 관리한다(커밋/롤백 시 자동 해제). 단, 락 보유가 트랜잭션 길이에 묶이고, `Order` 같은 공통 자원의 락 키 일관성이 코드에 흩어진 채로 남는다.
- **비관 락 유지 + 트랜잭션 범위 축소** — 외부 호출(PG 등)을 트랜잭션 밖으로 빼 락 보유 시간을 줄인다. 인프라 변화 없이 "락 범위가 길다" 문제를 직접 해소한다. 단, 도메인별 락 산재와 키 일관성 문제는 그대로다.
- **Redis 분산 락(Redisson)** — 이미 운영 중인 Redis로 키 단위 잠금. 락 획득·타임아웃·범위를 코드에서 명시적으로 제어하고, DB 커넥션을 잠금 대기로 점유하지 않는다. 공통 락 키 규칙으로 도메인 간 직렬화 기준을 한곳에 고정할 수 있다. 단, 락과 DB 트랜잭션이 분리되어 경계를 직접 관리해야 하고, Redis 장애가 락 가용성에 직접 영향을 준다.

## 결정

레코드 상태 변경 동시성 제어는 Redis 분산 락으로 통일한다. Redisson을 도입하고, `Order`·`Payment`·`RefundRequest`의 `PESSIMISTIC_WRITE` 조회를 분산 락으로 대체한다.

`Product.@Version` 낙관적 락(ADR-001)은 그대로 둔다. **분산 락은 낙관 락을 대체하지 않는다.** 역할 경계는 다음과 같다.

- **생성 시 중복 차단** = 낙관 락(`Product.@Version`) — ADR-001
- **기존 레코드 상태 변경 직렬화** = Redis 분산 락 — 본 ADR(기존 비관 락 대체)

**공통 자원으로서의 `Order` 락(공통 컨벤션).** `Order` row를 잠그는 모든 도메인(주문·결제·환불·리뷰)은 **동일한 락 키 `lock:order:{orderId}`** 를 공유한다. 도메인마다 다른 키를 쓰면 같은 주문에 대한 작업이 서로 직렬화되지 않으므로, 이 키 규칙은 특정 도메인 소유가 아니라 프로젝트 공통 규칙으로 고정한다. 리뷰(`ReviewService`)도 자기 자원 락 없이 이 공통 키만 사용한다.

**결제 멱등키와의 구분.** 결제 중복 차단은 멱등키(`findByIdempotencyKey`)가 담당하고, 분산 락은 결제·주문 상태 전이 직렬화를 담당한다. 둘은 역할이 다르며, 분산 락이 멱등키를 대체하지 않는다.

**채팅방은 본 결정 범위에서 제외한다.** `ChatRoom` 비관 락(`sendMessage`의 `findWithParticipantsByIdForUpdate`)은 메시지 전송마다 잠그는 빈번한 경로이고, 목적이 `lastMessageAt` 갱신이라 강한 직렬화가 필요하지 않으며, 채팅 메시지 저장 비동기화(`@Async`) 방향과도 상충한다. 채팅방 락은 분산 락으로 옮기지 않고, 비관 락 유지 또는 제거를 별도 과제로 다룬다.

"비관 락 유지" 안은 공통 자원의 직렬화 기준을 한곳에 모으지 못한다는 점에서 제외한다. "트랜잭션 범위 축소"는 본 결정과 배타적이지 않으며, 분산 락 도입과 함께 적용할 수 있는 개선으로 남긴다.

`docs/CONVENTIONS.md`의 "Redis 분산락은 신중하게만 제안한다"는 *근거 없는 도입*을 막으려는 가이드이며, 본 ADR이 그 근거가 된다.

## 결과

- `build.gradle`에 Redisson 의존성을 추가한다. 로컬 실행 전제에 Redis가 포함된다(이미 검색 캐시로 필요).
- **공통 락 키 규칙을 정의한다.** `Order`는 `lock:order:{orderId}`로 주문·결제·환불·리뷰가 공유한다. `Payment`·`RefundRequest`는 각 자원 단위 키를 정의한다. 같은 레코드를 보호하는 모든 경로가 동일한 키를 쓰도록 한다.
- 락과 DB 트랜잭션의 경계를 명확히 한다: 락 획득 → 트랜잭션 시작·커밋 → 락 해제 순서를 지키고, 락 TTL(임대 시간)은 트랜잭션 최대 수행 시간보다 길게 잡는다.
- 락 획득 실패(타임아웃)와 Redis 장애 시 동작을 정의한다. 락을 얻지 못하면 상태 변경을 진행하지 않고 실패로 처리한다.
- 전환이 끝나면 `Order`·`Payment`·`RefundRequest`의 `findById...ForUpdate` 계열 메서드와 `@Lock`을 제거한다. **`ChatRoom`의 비관 락은 이 범위에 포함하지 않는다.**
- 기존 비관 락에 의존하는 테스트(`PaymentServiceTest`, `RefundServiceDataIntegrityTest` 등)를 분산 락 기준으로 갱신한다.
- 동시성 테스트는 같은 주문에 대한 동시 작업(결제·취소·확정·환불·리뷰)이 하나만 성공하거나 직렬화됨을 검증한다.
