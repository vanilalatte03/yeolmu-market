# ADR-005: 모의 안전결제와 거래 상태 정책

- 상태: 채택됨
- 날짜: 2026-06-24
- 관련: `docs/PRD.md` (주문, 결제, 리뷰), `docs/API.md` (주문 API, 결제 API, 리뷰 API), `docs/ERD.md` (orders, payment, review)

## 맥락

열무마켓은 1:1 중고거래 중개 플랫폼이다.
주문, 결제, 배송 증빙, 구매확정, 환불, 분쟁을 어떤 단계에서 어떤 의미로 다룰지 명확히 정하지 않으면 문서와 구현에서 다음 문제가 생긴다.

- 플랫폼이 실제 결제대금을 보관하거나 정산하는 것처럼 오해될 수 있다.
- 결제 전 취소, 배송 후 환불 요청, 구매확정, 분쟁의 경계가 흐려진다.
- P0 주문 예약 흐름과 P1 모의 안전결제 흐름이 섞여 구현 범위가 커진다.
- P2 리뷰와 분쟁 흐름이 P1 결제 취소 흐름에 잘못 포함될 수 있다.

따라서 실제 PG/에스크로 연동 없이 프로젝트 안에서 표현할 거래 상태와 제외할 범위를 확정한다.

## 검토한 대안

- **P0 예약 주문 + P1 모의 안전결제 + P2 환불/분쟁** — 프로젝트 범위를 단계별로 유지하면서 거래 상태를 명확히 표현한다. 실제 돈의 보관, 에스크로, 정산은 구현하지 않는다.
- **실제 PG/에스크로 연동** — 실제 서비스에 가깝지만 외부 사업자 계약, 결제 검증, 정산, 환불, 보안 요구사항이 커져 현재 프로젝트 범위를 넘는다.
- **결제 후 취소를 즉시 환불로 단순 처리** — 구현은 단순하지만 배송 후 문제 제기와 판매자 거절, 분쟁 보류 상태를 표현할 수 없다.

## 결정

열무마켓은 1:1 중고거래 중개 플랫폼으로 정의한다.
P1부터 모의 안전결제 흐름을 제공하되, 플랫폼이 직접 돈을 보관한다고 표현하지 않는다.
실제 결제대금 보관, 에스크로, 판매자 송금은 PG/에스크로 사업자 영역으로 본다.

이번 프로젝트에서는 다음을 구현하지 않는다.

- 실제 PG 연동
- 플랫폼 수수료와 PG 수수료
- 실제 판매자 정산과 계좌 송금
- 부분 환불
- 법적 분쟁 중재

구매확정은 "거래 완료 및 판매자 지급 가능 상태"를 의미한다.
P1에서는 실제 송금 없이 판매자 지급 가능 상태만 표현한다.

### P0 범위

P0에서는 결제 없는 주문 예약 흐름만 구현한다.

- 구매자가 주문을 생성하면 상품은 예약 중 상태가 된다.
- 주문 생성은 구매자의 거래 의사 확정으로 본다.
- 동일 상품에는 성공 주문이 하나만 생성되어야 한다.
- 구매자는 P0 주문을 취소할 수 있다.
- 주문이 취소되면 상품은 다시 판매 중 상태가 된다.
- 결제, 배송 증빙, 구매확정, 환불, 분쟁은 P0에서 구현하지 않는다.

### P1 범위

P1에서는 모의 안전결제를 구현한다.

- 하나의 주문에는 성공 결제를 한 번만 허용한다.
- 같은 결제 요청이 반복되어도 멱등키로 중복 결제를 막는다.
- 결제 성공 시 주문은 결제 완료 상태가 된다.
- 결제 실패 또는 결제 전 취소 시 주문은 취소 상태가 되고 상품은 판매 중으로 돌아간다.
- 배송 증빙 등록 전 결제 후 취소 시 주문은 환불 상태가 되고 상품은 판매 중으로 돌아간다.
- 판매자는 결제 완료된 주문에 배송 증빙을 등록할 수 있다.
- 배송 증빙은 송장 번호로 구성한다.
- 판매자가 배송 증빙을 등록하면 주문은 배송 중 상태가 된다.
- 배송 증빙 등록 후 구매자의 단순 구매취소 버튼은 비활성화된다.
- 배송 증빙 등록 후 구매자 단순 취소는 서버에서도 거부한다.
- 구매자가 물건을 받고 문제가 없다고 판단하면 구매확정을 할 수 있다.
- 구매확정 시 주문은 거래 완료 상태가 된다.
- 구매확정 시 상품은 판매 완료 상태가 된다.

### P2 범위

P2에서는 환불 요청과 분쟁 흐름을 구현한다.

- 배송 증빙 등록 후 구매자가 거래를 취소하려면 단순 취소가 아니라 환불 요청을 해야 한다.
- 판매자가 환불 요청을 승인하면 주문은 환불 상태가 된다.
- 환불 요청이 승인되면 상품은 별도 반품 확인 없이 자동으로 판매 중 상태가 된다.
- 판매자가 환불 요청을 거절하면 주문은 분쟁 상태가 된다.
- 환불 요청 상태는 `REQUESTED -> APPROVED`, `REQUESTED -> DISPUTED`, `DISPUTED -> CLOSED` 흐름만 사용한다.
- 판매자 거절은 별도 거절 상태를 두지 않고 곧바로 `DISPUTED`로 전환한다.
- 분쟁 상태에서는 구매확정과 판매자 지급 가능 처리를 보류한다.
- 분쟁은 구매자 환불 또는 판매자 지급 가능 처리 중 하나로 종료한다.
- 분쟁이 구매자 환불로 종료되면 상품은 별도 반품 확인 없이 자동으로 판매 중 상태가 된다.
- 거래 완료된 주문에 대해 리뷰와 평점을 작성할 수 있다.
- 구매자와 판매자는 같은 주문의 상대방에게 각각 한 번씩 리뷰할 수 있다.

### 상태 흐름

P0 주문 상태 흐름:

```text
CREATED -> CANCELED
```

P1 주문 상태 흐름:

```text
CREATED -> PAID
CREATED -> CANCELED
PAID -> SHIPPING
SHIPPING -> COMPLETED
PAID -> REFUNDED
```

P2 주문 상태 흐름:

```text
SHIPPING -> REFUND_REQUESTED
REFUND_REQUESTED -> REFUNDED
REFUND_REQUESTED -> DISPUTED
DISPUTED -> REFUNDED
DISPUTED -> COMPLETED
```

상태 의미는 다음 기준을 따른다.

- 취소: 판매자가 보내기 전 거래를 없던 일로 돌리는 것
- 환불 요청: 판매자가 보낸 뒤 문제가 있어 돈을 돌려달라고 요청하는 것
- 환불 승인: 상품을 별도 반품 확인 없이 판매 중 상태로 자동 전환하는 것
- 분쟁: 환불 요청에 대해 구매자와 판매자가 합의하지 못한 상태
- 구매확정: 구매자가 물건 수령과 이상 없음을 확인해 거래를 완료하는 것

## 결과

- PRD는 P0 주문 예약, P1 모의 안전결제/배송 증빙/구매확정, P2 환불 요청/분쟁/양방향 리뷰 범위가 드러나도록 정리해야 한다.
- API 문서는 `OrderStatus`에 `SHIPPING`, `REFUND_REQUESTED`, `DISPUTED`를 추가하고, 배송 증빙 등록, 환불 요청, 환불 승인/거절, 분쟁 종료 API를 단계별 범위에 맞게 정의해야 한다.
- 기존 결제 취소 API는 배송 전 취소와 배송 후 환불 요청을 구분하도록 재검토해야 한다.
- ERD는 배송 증빙 송장 번호, 환불 요청/분쟁 상태에 필요한 컬럼 또는 테이블, 양방향 리뷰 제약을 반영해야 한다.
- 문서와 UI 문구에서 플랫폼이 돈을 보관, 에스크로, 정산, 송금한다고 표현하지 않는다.
- 구현 이슈는 P0, P1, P2 단계가 섞이지 않도록 분리한다.
