# ADR-011: 검색 목록 캐시와 상품 표시 캐시 분리

- 상태: 채택됨
- 날짜: 2026-06-29
- 관련: ADR-002(Redis 검색 캐시 — 본 ADR이 그 캐시 구조와 무효화 방식을 구체화한다)

## 맥락

ADR-002에 따라 상품 검색 결과를 Redis에 캐싱한다. 현재는 검색 조건을 키로 `PageResponse<SearchProductResponse>`(상품 표시 내용까지 포함한 완성된 페이지)를 통째로 캐싱하고, 변경이 생기면 `cache.invalidate()`로 캐시 전체를 비운다.

목표는 상품 변경 시 무관한 캐시까지 전부 날리지 않는 것이다. 처음에는 변경된 productId가 포함된 캐시만 역인덱스로 부분 무효화하려 했으나, 이는 검색 목록 캐시에 대해 **원리적으로 불완전하다.**

검색 목록은 "조건 + 페이지"별 결과 집합이고, 한 상품의 변경은 세 가지로 캐시에 영향을 준다.

1. **표시값 변경** — 그 상품이 들어 있는 캐시의 제목·가격 등이 바뀐다.
2. **membership 변경** — 신규 등록, `RESERVED`→`ON_SALE`, 제목·설명 변경으로 새 키워드에 매칭되어, 그 상품이 *없던* 검색 결과에 새로 진입한다.
3. **order/page 변경** — 가격 변경으로 정렬 위치가 바뀌거나, 숨김·삭제로 뒤 상품들이 앞 페이지로 당겨진다.

productId 역인덱스는 "이미 그 상품이 들어 있던 캐시"만 찾을 수 있어 1번만 해결한다. 2·3번은 변경 대상 캐시에 그 productId가 들어 있지 않으므로 역인덱스로 찾지 못한다. 어떤 검색 조건·페이지가 이 변경에 영향받는지는 쿼리를 다시 실행해야만 알 수 있어, element(productId) 단위 무효화로는 검색 목록의 정합성을 보장할 수 없다.

또한 현재 캐시에는 상품 표시 내용이 통째로 들어 있어, 판매자 닉네임처럼 외부 엔티티에 의존하는 값이 박혀 캐시된다. 이 때문에 닉네임 변경(`UserService`)도 검색 캐시를 무효화해야 하고, 그 변경은 productId 하나로 특정되지 않는다. 한편 사용자별 찜 정보(`wished`·`wishCount`)는 이미 캐시에 넣지 않고 `SearchService.withWishSummaries`가 캐시 결과의 productId를 배치로 조립한다. **"캐시에는 안정 데이터만, 가변·외부 의존 데이터는 캐시 밖 조립"**이라는 패턴이 이미 부분적으로 동작한다.

## 검토한 대안

- **전체 무효화 유지(현행)** — 단순하다. 변경이 잦으면 캐시 효용이 낮고, "어떤 변경이 어떤 캐시에 영향을 주는가"가 코드에 드러나지 않는다.
- **productId 역인덱스 부분 무효화** — 표시값 변경에는 정확하지만, membership·order·page 변경을 잡지 못한다. 신규 진입·정렬 이동·페이지 밀림에서 stale이 남아 정합성을 보장하지 못하므로 제외한다.
- **TTL 단축** — stale 허용 시간만 줄인다. 정밀도와 무관하며, 짧은 TTL은 캐시 효용을 떨어뜨린다.
- **목록 캐시와 표시 캐시 분리(채택)** — 정밀 무효화가 불가능한 "목록 구성"은 경량으로 만들어 버전으로 전역 무효화하고, 무거운 "표시 데이터"는 productId 단위로 정밀하게 무효화·조립한다.

## 결정

검색 캐시를 두 층으로 분리한다.

**1. 검색 목록 캐시 — 무엇이 어떤 순서로**

- 키는 `검색 조건 + searchIndexVersion`, 값은 `productId 목록 + 페이지 메타데이터`(전체 건수·페이지 수 등)다. 상품 표시 내용은 담지 않는다.
- 목록 구성에 영향을 주는 변경(등록·삭제·숨김·상태·가격·제목·설명)은 `searchIndexVersion`을 증가시켜 **전역 무효화**한다. 옛 버전 키는 더 이상 참조되지 않고 TTL로 정리된다.
- **버전 조회는 Redis 장애에 안전해야 한다.** 버전을 캐시 키에 넣으므로, 버전 조회 실패가 키 생성(SpEL) 단계에서 예외가 되면 Spring Cache의 캐시 미스→MySQL 폴백 경로를 타지 못한다(Spring의 `CacheErrorHandler`는 캐시 get/put/evict 연산 실패만 처리하고, 키 생성 예외는 그대로 전파된다). 따라서 버전 제공 컴포넌트(예: `SearchIndexVersionProvider`)는 Redis 예외를 자체적으로 잡아 폴백 키를 반환해 캐시를 우회(MySQL 직행)하도록 한다.

**2. 상품 표시 데이터 — 어떻게 보일지**

- 목록 캐시가 돌려준 productId로 표시 데이터를 조립한다.
  - `productIds → 상품 projection`. projection 범위는 `title`·`price`·`status`·`thumbnailUrl`·`sellerId`·`createdAt`이다. `sellerNickname`은 제외(아래 닉네임 조립), `wishCount`·`wished`는 제외(찜 조립), `description`은 검색 조건(키워드 매칭)에만 쓰고 응답·projection에는 담지 않는다. productId 단위로 정밀 무효화할 수 있다(필요 시 productId 단위 캐시).
  - `sellerIds → 닉네임 배치 조회`. 닉네임은 목록 캐시에 넣지 않는다.
  - `productIds → 찜 배치 조회`(현행 `withWishSummaries` 유지).
- 표시값 변경은 해당 productId의 표시 데이터만 무효화하면 된다. 이 층은 단일 상품 데이터라 membership·order 문제가 없다.

**두 변경 층은 배타적이지 않다.** 제목·가격·상태 변경은 목록 구성(키워드 매칭·정렬·노출)과 표시값을 동시에 바꾼다. 이런 변경은 `searchIndexVersion` 증가와 해당 productId 표시 데이터 무효화를 **모두** 수행한다. 썸네일처럼 표시값만 바뀌는 변경은 productId 표시 무효화만 한다. 등록·삭제·숨김은 목록 구성 변경이며, 삭제·숨김은 표시 데이터도 함께 정리한다.

**이벤트는 의도별로 둘로 나눈다.** 단일 이벤트에 플래그를 넣기보다, 목록 인덱스 변경(예: `ProductSearchIndexChangedEvent(productId)`)과 표시 데이터 변경(예: `ProductDisplayChangedEvent(productId)`)으로 분리해 변경 의도를 코드에 드러낸다. 제목·가격·상태 변경은 두 이벤트를 모두 발행한다. 기존 `ProductSearchCacheEvictionEvent`(빈 record)는 이 둘로 대체한다.

**원칙.** 검색 캐시에는 목록 구성과 상품 고유 데이터만 둔다. 가변·개인화·외부 엔티티 의존 데이터(찜·판매자 닉네임)는 캐시 밖에서 배치 조립한다. 닉네임을 분리하므로 `UserService`의 검색 캐시 무효화 발행은 제거한다.

productId 역인덱스 안은 membership·order·page 변경에서 정합성을 보장하지 못하므로 채택하지 않는다.

## 결과

- **캐시 구조 변경**: 검색 목록 캐시는 `PageResponse<SearchProductResponse>`(표시 완성본) 대신 productId 목록 + 페이지 메타로 바꾼다. 응답은 목록 캐시 + 표시 조립으로 만든다.
- **searchIndexVersion 도입**: 전역 버전 카운터(예: Redis `search:index:version`)를 두고 목록 구성 변경 시 증가시킨다. 캐시 키에 버전을 포함하며, 버전 조회 실패 시 폴백 키로 캐시를 우회한다(위 "버전 조회는 Redis 장애에 안전해야 한다").
- **무효화 트리거 재정의**(두 층이 겹칠 수 있음):
  - 목록 구성 변경(등록·삭제·숨김·상태·가격·제목·설명) → `searchIndexVersion` 증가.
  - 표시값 변경(가격·제목·상태·썸네일 등) → 해당 productId 표시 데이터 무효화.
  - 제목·가격·상태처럼 둘 다에 해당하는 변경 → 두 처리를 모두 수행(두 이벤트 발행).
  - `UserService` 닉네임 변경 발행은 제거(조립 시점에 최신 닉네임을 채운다).
- **재생성 비용**: 버전 증가로 목록 캐시가 무효화되면 검색 쿼리와 count 쿼리는 여전히 다시 실행해야 한다. 다만 seller join·썸네일 조회·응답 DTO 직렬화 부담이 줄고, 버전과 무관한 표시 projection 캐시를 재사용할 수 있어 전체 페이지를 통째로 재구성하는 것보다 가볍다.
- **TTL 분리 = 최신성/효율 안전망**: 목록 캐시는 기본 30초(`list-ttl`)로 짧게 유지해, 버전 증가 이벤트가 빗나가거나 누락돼도 membership·order·page stale이 빠르게 수렴하게 한다. 표시 캐시는 productId 단위 무효화가 가능하고 버전과 무관하게 재사용되므로 기본 5분(`display-ttl`)을 유지해 hit ratio를 확보한다. 두 TTL은 `application.yml`로 외부화한다.
- 무효화는 ADR-002대로 AFTER_COMMIT에 동작한다.
- **테스트**: 표시값 변경뿐 아니라 (1) 신규 등록·`RESERVED`→`ON_SALE`로 검색 결과에 새로 진입, (2) 가격 변경에 따른 정렬 이동, (3) 숨김·삭제에 따른 페이지 밀림에서 다음 검색이 최신 구성을 반환하는지, (4) 닉네임 변경 후 최신 닉네임이 조립되는지, (5) Redis 버전 조회 실패·캐시 장애 시 MySQL 폴백이 동작하는지를 검증한다. 기존 "무관한 검색 캐시는 유지" 검증은 폐기한다.
