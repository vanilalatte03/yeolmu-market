# 열무마켓 PRD 

## 1. 목적

열무마켓은 사용자가 중고 상품을 등록하고, 원하는 상품을 검색하고, 판매자와 실시간 채팅으로 거래를 협의한 뒤 주문으로 거래를 확정하는 중고거래 서비스다.

이 PRD의 목적은 2주 안에 구현할 제품 범위와 성공 기준을 정하는 것이다. 구현 방식, 화면 밖 세부 설계, 데이터 구조, 기술 선택 이유는 별도 기술 메모에서 관리한다.

## 2. 프로젝트 목표

열무마켓은 많은 기능을 얕게 구현하는 서비스가 아니다. 다음 3가지 경험이 실제로 동작하는 것을 목표로 한다.

1. 사용자는 상품을 등록하고 검색해서 거래 후보를 찾을 수 있다.
2. 구매자와 판매자는 실시간 채팅으로 거래를 협의할 수 있다.
3. 동일 상품에 여러 구매자가 동시에 주문해도 하나의 거래만 확정된다.

## 3. 성공 기준

| 구분 | 성공 기준 |
|---|---|
| 거래 흐름 | 회원가입, 로그인, 상품 등록, 상품 검색, 채팅, 주문까지 이어지는 핵심 흐름이 동작한다. |
| 동시 주문 | 하나의 판매 중 상품에 동시에 여러 주문이 들어와도 성공 주문은 1건만 생긴다. |
| 검색 성능 | 같은 조건의 상품 검색에서 반복 조회 성능 개선을 확인할 수 있다. |
| 실시간 채팅 | 구매자와 판매자가 실시간으로 메시지를 주고받고 이전 메시지를 다시 볼 수 있다. |
| 우선순위 준수 | P0 범위가 완료되기 전에는 P1, P2 기능을 우선하지 않는다. |

## 4. 사용자

| 사용자 | 설명 |
|---|---|
| 비회원 | 상품 목록과 상세를 볼 수 있고 회원가입 또는 로그인을 할 수 있다. |
| 회원 | P0: 상품 등록, 검색, 채팅, 주문. P1: 찜. P2: 리뷰를 사용할 수 있다. |
| 관리자 | P0: 부적절한 상품 숨김 처리. P1: 카테고리 관리를 할 수 있다. |

관리자 전용 화면은 만들지 않는다. 관리자 기능은 권한이 있는 사용자만 접근 가능한 관리 기능으로 제공한다.

## 5. 범위

### P0: 필수 범위

P0는 프로젝트의 최소 완성 범위다. 일정이 부족해도 제외하지 않는다.

- 회원가입, 로그인, 로그아웃, 내 정보 조회와 수정
- 상품 등록, 수정, 삭제, 숨김, 목록 조회, 상세 조회
- 상품 검색
- 반복 검색 성능 개선 확인
- 인기 검색어 조회
- 주문 생성, 조회, 취소와 동시 주문 방지
- 구매자와 판매자 간 실시간 채팅
- 채팅 메시지 저장과 이전 메시지 조회

### P1: 확장 범위

P1은 P0 완료 후 구현한다.

- 모의 결제
- 결제 중복 방지
- 구매확정(거래 완료)
- 상품 이미지 업로드
- 카테고리
- 찜

### P2: 선택 범위

P2는 일정이 남을 때만 구현한다.

- 리뷰와 평점
- 상품 조회수

## 6. 핵심 사용자 흐름

### 6.1 판매자 흐름

1. 회원가입 또는 로그인한다.
2. 상품명, 설명, 가격을 입력해 상품을 등록한다. 카테고리와 상품 이미지는 P1에서 추가한다.
3. 구매 희망자와 채팅으로 거래를 협의한다.
4. 구매자의 주문이 생성되면 상품은 예약 상태가 된다.
5. 거래가 완료되면 상품은 판매 완료 상태가 된다. (구매확정/거래 완료는 P1)

### 6.2 구매자 흐름

1. 회원가입 또는 로그인한다.
2. 상품을 검색하거나 목록에서 상품을 탐색한다.
3. 판매자에게 채팅을 보낸다. 관심 상품 찜은 P1에서 추가한다.
4. 구매 의사가 확정되면 주문을 생성한다.
5. 거래 완료 후 판매자에게 리뷰를 남기는 기능은 P2에서 추가한다.

### 6.3 관리자 흐름

1. 부적절한 상품을 숨김 처리한다. (P0)
2. 카테고리를 관리한다. (P1)

## 7. 기능 요구사항

### 7.1 회원

- 사용자는 이메일, 비밀번호, 닉네임으로 가입할 수 있다.
- 사용자는 이메일과 비밀번호로 로그인할 수 있다.
- 로그인한 사용자는 로그아웃할 수 있다.
- 로그인한 사용자는 자신의 회원 정보를 조회할 수 있다.
- 로그인한 사용자는 자신의 닉네임 또는 비밀번호를 수정할 수 있다.
- 로그인하지 않은 사용자는 보호된 기능을 사용할 수 없다.
- 사용자는 다른 사용자의 상품, 주문, 채팅, 리뷰를 임의로 수정할 수 없다.

### 7.2 상품

- 회원은 상품을 등록할 수 있다.
- 상품 등록 시 상품명, 설명, 가격은 필수다.
- 회원은 자신이 등록한 상품만 수정하거나 삭제할 수 있다.
- 삭제된 상품은 일반 사용자에게 노출되지 않는다.
- 관리자는 상품 노출 상태를 숨김으로 변경할 수 있다.
- 숨김 상품은 일반 목록과 검색 결과에 노출되지 않는다.
- 상품 이미지 업로드는 P1 범위다. 한 상품에 여러 이미지를 등록할 수 있다.
- 상품은 판매 중, 예약 중, 판매 완료, 삭제 상태를 가진다.
- 상품 노출 상태는 보임, 숨김으로 별도 관리한다.

### 7.3 상품 검색

- 사용자는 키워드로 상품을 검색할 수 있다.
- 사용자는 가격 범위, 상품 상태로 검색 결과를 좁힐 수 있다. (카테고리 필터는 P1)
- 검색 결과는 페이지 단위로 조회할 수 있다.
- 캐시 적용 전후의 검색 결과는 같은 조건에서 동일해야 한다.
- 같은 조건을 반복 조회할 때 캐시로 성능 개선을 확인할 수 있어야 한다.
- 사용자는 인기 검색어 상위 목록을 조회할 수 있다.

### 7.4 주문

- 회원은 판매 중인 상품을 주문할 수 있다.
- 회원은 자신의 상품을 주문할 수 없다.
- 이미 예약 중이거나 판매 완료된 상품은 주문할 수 없다.
- 동일 상품에 동시에 여러 주문이 들어와도 하나의 주문만 성공해야 한다.
- 주문이 성공하면 상품은 예약 중 상태가 된다.
- 회원은 자신이 구매자 또는 판매자로 참여한 주문을 조회할 수 있다.
- 주문은 생성, 결제 완료, 거래 완료, 취소, 환불 상태를 가진다.
- P0 범위에서는 주문 생성, 주문 상세/목록 조회, 취소를 다룬다. 결제 완료, 거래 완료, 환불은 결제(P1)와 함께 구현한다.
- 허용되지 않은 주문 상태 변경은 실패해야 한다.

### 7.5 채팅

- 구매자는 상품 판매자에게 채팅을 시작할 수 있다.
- 동일 상품에 대해 같은 구매자와 판매자 사이의 채팅방은 하나만 유지한다.
- 채팅방 참여자만 메시지를 보내고 조회할 수 있다.
- 메시지는 실시간으로 상대방에게 전달되어야 한다.
- 보낸 메시지는 저장되어 이후 다시 조회할 수 있어야 한다.
- 이전 메시지는 최신 메시지 기준으로 순차 조회할 수 있어야 한다.

### 7.6 결제[p1]

- 결제는 실제 결제사 연동 없이 성공, 실패, 취소를 시뮬레이션한다.
- 하나의 주문에는 성공 결제가 한 번만 허용된다.
- 같은 결제 요청이 반복되어도 중복 결제가 발생하지 않아야 한다.
- 결제 성공 시 주문은 결제 완료 상태가 된다.
- 결제 실패 또는 결제 전 취소 시 상품은 다시 판매 중 상태가 된다.
- 결제 후 취소는 환불 상태로 처리한다.

### 7.7 카테고리[p1]

- 사용자는 카테고리 목록을 조회할 수 있다.
- 판매자는 상품 등록 시 카테고리를 선택할 수 있다.
- 사용자는 카테고리별 상품을 조회할 수 있다.
- 관리자는 카테고리를 생성, 수정, 삭제할 수 있다.

### 7.8 찜[p1]

- 회원은 상품을 찜할 수 있다.
- 회원은 찜한 상품을 취소할 수 있다.
- 회원은 자신이 찜한 상품 목록을 조회할 수 있다.
- 상품별 찜 수를 확인할 수 있다.

### 7.9 리뷰와 평점[p2]

- 거래가 완료된 주문의 구매자만 판매자에게 리뷰를 작성할 수 있다.
- 리뷰는 평점과 내용을 포함한다.
- 평점은 1점부터 5점까지 허용한다.
- 사용자는 자신이 작성한 리뷰와 자신이 받은 리뷰를 조회할 수 있다.
- 리뷰 작성자만 자신의 리뷰를 수정하거나 삭제할 수 있다.
- 판매자 평점은 받은 리뷰를 기준으로 계산된다.

### 7.10 상품 조회수[p2]

- 상품 상세 조회 시 조회수가 증가한다.
- 사용자는 상품 상세에서 조회수를 확인할 수 있다.

## 8. 상태 요구사항

### 8.1 상품 상태

| 상태 | 의미 |
|---|---|
| 판매 중 | 주문 가능한 상태 |
| 예약 중 | 주문이 생성되어 다른 사용자가 주문할 수 없는 상태 |
| 판매 완료 | 거래가 완료된 상태 |
| 삭제 | 판매자가 삭제한 상태 |

### 8.2 상품 노출 상태

| 상태 | 의미 |
|---|---|
| 보임 | 일반 사용자에게 노출되는 상태 |
| 숨김 | 관리자가 일반 목록과 검색 결과 노출을 막은 상태 |

### 8.3 주문 상태

| 상태 | 의미 |
|---|---|
| 생성 | 주문이 생성되고 상품이 예약된 상태 |
| 결제 완료 | 결제가 완료된 상태 |
| 거래 완료 | 구매확정이 끝난 상태 |
| 취소 | 결제 전 주문이 취소된 상태 |
| 환불 | 결제 후 거래가 취소된 상태 |

P0에서는 생성, 취소만 사용한다. 결제 완료, 거래 완료, 환불은 결제(P1)와 함께 도입한다.

## 9. 제외 범위

- 모바일 앱
- 관리자 백오피스 화면
- 실제 결제사 연동
- 정산과 판매자 대금 지급
- 배송 추적, 송장 등록, 택배사 연동
- 포인트, 쿠폰
- 로그인 연장용 재발급 토큰
- 소셜 로그인
- 신고, 차단
- 추천 알고리즘

## 10. 출시 전 확인 시나리오

P0 완료 여부는 다음 시나리오로 판단한다.

1. 사용자가 가입하고 로그인한다.
2. 판매자가 상품을 등록한다.
3. 구매자가 상품을 검색하고 상세를 조회한다.
4. 구매자가 판매자와 채팅을 시작한다.
5. 구매자와 판매자가 실시간 메시지를 주고받는다.
6. 구매자가 상품을 주문한다.
7. 동일 상품에 동시에 여러 주문을 시도해도 하나만 성공한다.
8. 동일 조건 반복 검색의 성능 개선을 확인한다.

P1 완료 시 결제, 카테고리, 찜 흐름을 추가로 확인한다. P2 완료 시 리뷰, 평점, 조회수 흐름을 추가로 확인한다.
